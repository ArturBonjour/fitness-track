# ВВЕДЕНИЕ

Стремительная цифровизация сферы здоровья и физической активности обусловила формирование устойчивого рынка программных решений, призванных помочь пользователю в планировании тренировок, контроле прогресса и принятии решений на основе структурированных данных. Аналитики Grand View Research оценивают объем мирового рынка мобильных приложений для здоровья и фитнеса в 14,7 млрд долларов США при среднегодовом темпе роста 17,6% в горизонте 2025–2030 годов [17]. Вместе с тем значительная часть представленных на рынке сервисов страдает функциональной перегруженностью или ориентирована исключительно на узкие сценарии использования, оставляя незаполненной нишу легких универсальных трекеров.

Степень разработанности темы в области отечественных исследований остается невысокой: большинство российских разработчиков применяют готовые зарубежные платформы, не адаптированные к специфике отечественного пользователя. Зарубежные решения — Strava, MyFitnessPal, Google Fit — занимают лидирующие позиции, однако исследования практик самоотслеживания свидетельствуют о том, что пользователи прекращают работу с приложением при отсутствии комплексного инструментария для отслеживания целей и аналитики [9].

Информационные технологии становятся ключевым инструментом персонализации в сфере здоровья и спорта [4]. Проблемная область связана с тем, что пользовательские данные о тренировках и целях часто хранятся фрагментарно, а применяемые инструменты не обеспечивают достаточного баланса между простотой интерфейса и аналитическими возможностями. Практическая задача данной работы состоит в проектировании и реализации полнофункционального веб-приложения фитнес-трекера с современной клиент-серверной архитектурой.

Цель работы — разработать информационную систему персонального фитнес-мониторинга «Bogdanov FitTrack», обеспечивающую учет тренировок, постановку и отслеживание фитнес-целей, а также визуализацию пользовательского прогресса на основе структурированных данных, хранимых на сервере.

Для достижения поставленной цели необходимо решить следующие задачи.

1. Провести анализ предметной области персонального фитнес-мониторинга, выявить ключевые поведенческие паттерны цикличного взаимодействия пользователя с системой учета физической активности.
2. Осуществить сравнительный анализ ведущих программных аналогов (Strava, MyFitnessPal, Google Fit) и сформулировать функциональные и нефункциональные требования, устраняющие практический разрыв между существующими решениями.
3. Обосновать выбор архитектурного подхода, технологического стека и механизма аутентификации на основе требований безопасности, производительности и масштабируемости системы.
4. Спроектировать документно-ориентированную модель данных, спецификацию REST API из 22 маршрутов с JWT-защитой и компонентную иерархию React-приложения с применением UML-диаграмм.
5. Реализовать механизм JWT-аутентификации с защитой от перебора, полный цикл управления тренировками и фитнес-целями с автоматическим контролем прогресса, алгоритм расчета суточной нормы калорий по формуле Харриса-Бенедикта и механизм вычисления серии тренировочных дней.
6. Провести функциональное тестирование 44 сценариев использования системы с верификацией корректности обработки граничных значений, изоляции данных между пользователями и точности расчетных алгоритмов.

Объект исследования — процессы цифрового сопровождения персональной физической активности пользователя.

Предмет исследования — методы и средства проектирования и разработки веб-приложения фитнес-мониторинга с модульной клиент-серверной архитектурой, REST API и визуальной аналитикой.

Практическая значимость работы состоит в создании полностью функционирующего веб-приложения с открытым исходным кодом, пригодного для развертывания на любой платформе с поддержкой Node.js без лицензионных ограничений. Система применима в образовательной, корпоративной и спортивной среде как самостоятельный инструмент персонального фитнес-мониторинга. Методология тестирования (44 задокументированных тест-кейса), архитектурные решения и листинги ключевых алгоритмов могут использоваться как учебно-методические материалы при подготовке специалистов в области разработки веб-приложений. Возможности масштабирования включают интеграцию с носимыми устройствами, расширение базы типов тренировок и углубление аналитического блока.

Методы исследования, применяемые в работе: системный анализ предметной области; сравнительный анализ программных аналогов; метод прецедентов (use case) для выявления требований; объектно-реляционное и документно-ориентированное моделирование данных; компонентное проектирование пользовательского интерфейса; функциональное тестирование по методу «черного ящика».

Научная новизна работы заключается в обосновании и реализации механизма автоматического определения выполнения фитнес-цели, основанного на накопительном мониторинге количественного прогресса, а также алгоритма подсчета непрерывной серии тренировочных дней с применением множества временных меток. По нашему мнению, совместное использование данных алгоритмов в рамках единого клиент-серверного приложения создает качественно новый пользовательский опыт персонального фитнес-мониторинга.

Структурно работа включает введение, четыре главы основной части, заключение, список использованных источников и приложение А с ключевыми листингами кода. В первой главе рассматриваются теоретические и прикладные аспекты предметной области, формулируются функциональные и нефункциональные требования к системе. Во второй главе описывается проектирование системы: архитектура, модель данных, API и компонентная иерархия. В третьей главе представлена программная реализация и описание функциональных экранов. В четвертой главе проводится функциональное тестирование и автоматизированная верификация сборки.

# 1 АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ТРЕБОВАНИЙ К СИСТЕМЕ ФИТНЕС-ТРЕКИНГА

## 1.1 Предметная область персонального фитнес-мониторинга

Персональный фитнес-мониторинг представляет собой совокупность процессов планирования, фиксации и анализа тренировочной активности пользователя. Согласно рекомендациям Всемирной организации здравоохранения [11], взрослому человеку необходимо не менее 150 минут умеренной аэробной активности в неделю; цифровые инструменты фитнес-трекинга формируют информационную основу для выполнения данных нормативов на практике и формирования устойчивых поведенческих привычек.

С технической точки зрения ядро предметной области включает следующие процессы:
- календарное планирование тренировок;
- учет параметров тренировочной сессии (тип, дата, длительность, комментарии);
- постановку измеримых целей с дедлайнами;
- оценку текущего прогресса по количественным показателям;
- поддерживающую аналитику (графики, агрегированные показатели);
- персонализированные рекомендации на основе профиля и цели пользователя.

Ключевая особенность данной области — цикличность поведения пользователя. В отличие от разовых задач, фитнес-активность требует длительного сопровождения, регулярного ввода данных и мотивационной обратной связи. Следовательно, программный продукт должен быть ориентирован не только на хранение информации, но и на снижение когнитивной нагрузки при ежедневном использовании [10].

Для рассматриваемой области существенны следующие свойства информационной системы.

1. Оперативность ввода данных — пользователь должен фиксировать тренировку за минимальное количество действий.
2. Согласованность данных — изменения профиля, тренировок и целей должны немедленно отражаться в аналитике.
3. Наглядность прогресса — система должна предоставлять визуальные инструменты для интерпретации динамики показателей.
4. Надежность и безопасность — персональные данные должны быть защищены механизмами аутентификации и авторизации.
5. Расширяемость — архитектура должна допускать развитие функционала без переработки ядра системы.

Принципиальное значение имеет и аспект персонализации: исследования в области поведенческой экономики показывают, что индивидуализированная обратная связь значительно эффективнее поддерживает долгосрочную мотивацию к физической активности, чем обобщенные рекомендации. Именно этим обусловлено конкурентное преимущество персонализированных систем мониторинга перед универсальными решениями.

Таким образом, предметная область объединяет требования к удобству пользовательского интерфейса, согласованности данных, инженерной устойчивости и персонализационным возможностям программного решения.

Необходимо также отметить, что Google Fit прекратил поддержку на платформе Android в 2024 году, переведя пользователей на платформу Google Health Connect; данный факт дополнительно подчеркивает нестабильность бесплатной экосистемы сторонних сервисов и обосновывает разработку независимого решения.

По итогам анализа были выявлены типовые ограничения аналогов:
- недостаточная гибкость в управлении целями с количественными показателями прогресса;
- слабая связность между календарным планированием и аналитическим блоком;
- ограниченная персонализация интерфейса и поддержка темной темы;
- высокая зависимость REST API от платных тарифов при базовых потребностях разработчика.

Дополнительным фактором, ограничивающим применимость существующих решений в учебной и исследовательской среде, является закрытость исходного кода: ни один из рассмотренных аналогов не предоставляет возможности изучения архитектурных решений или адаптации под конкретные требования. Это обстоятельство делает разработку собственного решения с открытым кодом особенно актуальной.

Следовательно, разработка приложения «Bogdanov FitTrack» обоснована как с точки зрения функционального разрыва между существующими решениями и потребностями целевой аудитории, так и с позиции создания открытой платформы для персонального фитнес-мониторинга.

## 1.3 Формирование требований к разрабатываемой системе

На основе анализа предметной области и существующих решений сформирован набор требований к проектируемому приложению.

Система должна обеспечивать:
- регистрацию и аутентификацию пользователей с хранением персональных физических параметров;
- создание, просмотр, редактирование и удаление записей о тренировках;
- визуализацию тренировок в виде интерактивного месячного календаря;
- постановку фитнес-целей с начальным и целевым значениями, единицей измерения и дедлайном;
- автоматическое вычисление процента достижения цели и изменение статуса на «выполнена»;
- отображение графика динамики веса пользователя;
- агрегированную статистику: количество тренировок за неделю/месяц, серия дней, ИМТ;
- расчет суточной нормы калорий по формуле Харриса-Бенедикта с учетом уровня активности и цели;
- персонализированные рекомендации по тренировкам в зависимости от цели и уровня подготовки.

В таблице 1.2 представлены нефункциональные требования, которым должна соответствовать разрабатываемая система.

| Атрибут качества | Требование |
|---|---|

Анализ требований и современных практик веб-разработки позволяет обосновать следующий технологический стек.

Для серверной части выбран Node.js в сочетании с фреймворком Express.js [14]. Данное решение обеспечивает неблокирующую I/O-модель, широкую экосистему пакетов и простоту разработки REST API [7]. В качестве базы данных выбрана MongoDB с ODM-библиотекой Mongoose [19]: документно-ориентированная модель хранения наиболее органична для разнородных данных пользователей (профиль, история веса, тренировки, цели) [6].

Клиентская часть реализована на React — библиотеке декларативного построения UI-компонентов [23]. Навигация между страницами организована с применением React Router v6 [24]. Управление глобальным состоянием осуществляется через встроенный Context API без дополнительных зависимостей. Стилизация выполнена с применением Tailwind CSS — утилитарного фреймворка с поддержкой адаптивных префиксов и темной темы через стратегию `class` [26].

## 1.4 Выводы по первой главе

В первой главе проведен системный анализ предметной области персонального фитнес-мониторинга. Выявлено пять ключевых поведенческих паттернов взаимодействия пользователя с системой: планирование, учет активности, постановка целей, аналитика прогресса и обратная связь. ## 3.2 Реализация серверной части

Аутентификация реализована по модели Stateless JWT. При регистрации сервер: валидирует все входные поля через массив правил express-validator; проверяет уникальность email в коллекции `users`; хэширует пароль алгоритмом bcrypt с 10 раундами соли; сохраняет документ пользователя; возвращает JWT-токен со сроком 7 дней.

При входе применяется rate-limiter на базе библиотеки express-rate-limit [15]: не более 5 попыток за 15 минут с одного IP. Данный механизм защиты соответствует рекомендациям OWASP API Security Top 10 2023 по предотвращению атак перебора учетных данных (API2:2023 Broken Authentication) [22]. На рисунке 3.2 представлена реализация middleware JWT-верификации.

```javascript
// middleware/auth.js
module.exports = function (req, res, next) {
    let token = req.header('x-auth-token') || req.header('Authorization');
    if (token && token.startsWith('Bearer ')) token = token.slice(7);
    if (!token) return res.status(401).json({ message: 'Нет токена' });
    try {
        const decoded = jwt.verify(token, process.env.JWT_SECRET);
        req.user = decoded.user;
        next();
    } catch (err) {
        res.status(401).json({ message: 'Токен недействителен' });
    }
};
```

## 3.3 Реализация клиентской части

Хук `useAxiosInterceptor` настраивает перехватчик ответов Axios: при получении HTTP 401 токен удаляется из localStorage и пользователь перенаправляется на страницу входа. Это обеспечивает единое место обработки истечения сессии без дублирования логики в каждом компоненте.

Компонент `StatsWidget` — ключевая новая функция, аналогичная сводным дашбордам Strava и Nike Training Club. При монтировании загружает все тренировки пользователя и вычисляет следующие показатели.

1. Тренировок за неделю и суммарные минуты — фильтрация по `dayjs().startOf('isoWeek')` [13].
2. Тренировок за месяц и суммарные минуты — фильтрация по `dayjs().startOf('month')`.
3. Серия дней (streak) — обратный обход от сегодня по множеству дат (структура данных Set языка JavaScript [5]) с тренировками до первого пропуска.
4. Любимый тип тренировки за месяц — агрегация по полю `type` с выбором максимума.
5. ИМТ — вычисляется из данных профиля: `weight / (height/100)²`, с категоризацией по стандарту ВОЗ [27].

На рисунке 3.5 представлен алгоритм вычисления серии дней.

```javascript
// StatsWidget.js (фрагмент)
const datesSet = new Set(
    all.map(w => dayjs(w.date).format('YYYY-MM-DD'))
);
let streak = 0;
let d = dayjs();
while (datesSet.has(d.format('YYYY-MM-DD'))) {
    streak++;
    d = d.subtract(1, 'day');
}
```

Рисунок 3.5. Алгоритм вычисления серии тренировочных дней

Компонент `Calendar` реализует: генерацию сетки с учетом дня недели первого числа месяца (понедельник = 0); загрузку тренировок через `GET /api/workouts` с клиентской фильтрацией по текущему месяцу; цветные бейджи (фиолетовый — тренировка, зеленый — цель); открытие форм создания записей в модальном окне с предзаполнением выбранной даты.

Компонент CalorieCalculator реализует расчет суточной нормы калорий по формуле Харриса-Бенедикта [8]. Базальный уровень метаболизма (BMR) вычисляется по формулам (3.1) и (3.2) в зависимости от пола пользователя.

Для мужчин:

BMR = 88,362 + 13,397 × W + 4,799 × H − 5,677 × A,               (3.1)

Для женщин:

BMR = 447,593 + 9,247 × W + 3,098 × H − 4,330 × A,               (3.2)

где W — масса тела (кг); H — рост (см); A — возраст (лет).

Суточная норма калорий (TDEE) определяется как произведение BMR на коэффициент физической активности, после чего применяется поправочный множитель в зависимости от цели пользователя (похудение, поддержание, набор массы). На рисунке 3.6 представлен фрагмент реализации данного расчета.

```javascript
// CalorieCalculator.js (фрагмент)
let bmr;
if (user.gender === 'male') {
    bmr = 88.362 + 13.397*user.weight + 4.799*user.height - 5.677*user.age;
} else {
    bmr = 447.593 + 9.247*user.weight + 3.098*user.height - 4.330*user.age;
}
const tdee = bmr * activityMultipliers[formData.activityLevel];
const targetCalories = Math.round(tdee * goalMultipliers[formData.goal]);
```

# ЗАКЛЮЧЕНИЕ

В результате выполнения выпускной квалификационной работы спроектирована и реализована информационная система персонального фитнес-мониторинга «Bogdanov FitTrack». Созданное клиент-серверное веб-приложение обеспечивает полный цикл персонального фитнес-мониторинга: учет тренировок с визуализацией в интерактивном календаре, управление измеримыми целями с автоматическим отслеживанием прогресса, расчет суточного калоража и агрегированную аналитику. Поставленная цель работы достигнута в полном объеме: сформирована готовая к развертыванию программная система с открытым исходным кодом. Основные количественные показатели реализованного решения: 22 REST API-маршрута с JWT-защитой; 4 Mongoose-коллекции; 8 React-компонентов; 44 верифицированных тест-кейса (100% прохождение); 13 иллюстраций архитектурных и проектных решений; 16 сводных таблиц технических характеристик; поддержка разрешений от 320 до 1280 пикселей; расчет суточной нормы калорий по методике Харриса-Бенедикта [8] с двумя формулами для мужчин и женщин.

В процессе выполнения выпускной квалификационной работы последовательно решены следующие задачи.

1. Проведен анализ предметной области персонального фитнес-мониторинга: выявлены пять ключевых поведенческих паттернов (планирование, учет активности, постановка целей, аналитика, обратная связь) и пять критических свойств системы (оперативность ввода, согласованность данных, наглядность прогресса, надежность, расширяемость).

2. Осуществлен сравнительный анализ четырех ведущих аналогов (таблица 1.1), показавший, что ни один из них не предоставляет в бесплатном режиме полного комплекса: calendar-планирования, управления целями, калькулятора калорий и персонализированных рекомендаций одновременно. Сформированы 9 функциональных и 6 нефункциональных требований (таблица 1.2).

3. Обоснован технологический стек: Node.js/Express (неблокирующая I/O-модель), React (декларативная компонентная модель), MongoDB/Mongoose (гибкая документная схема), JWT (stateless-аутентификация). Обоснован механизм защиты от перебора учетных данных (rate-limiting, 5 попыток / 15 мин, рекомендация OWASP API Security Top 10 2023 [22]).

4. Спроектированы: трехуровневая архитектура (рисунок 2.1), документно-ориентированная модель данных из четырех коллекций (таблицы 2.1–2.3, рисунок 2.3), спецификация из 22 REST API-маршрутов (таблица 2.4), диаграмма последовательности JWT-авторизации (рисунок 2.4), диаграмма компонентов (рисунок 2.5) и диаграмма вариантов использования (рисунок 2.6).

Практическая значимость работы состоит в создании готовой программной основы с открытым исходным кодом (репозиторий: https://github.com/ArturBonjour/fitness-track), пригодной для внедрения и масштабирования в реальной среде. Разработанная система обеспечивает полный цикл персонального фитнес-мониторинга и может служить фундаментом для построения более широкой платформы здоровья.

```javascript
const mongoose = require('mongoose');

const UserSchema = new mongoose.Schema({
    name:     { type: String, required: true },
    email:    { type: String, required: true, unique: true },
    password: { type: String, required: true },
    gender:   { type: String, enum: ['male', 'female'], required: true },
    age:      { type: Number, min: 15, max: 100, required: true },
    weight:   { type: Number, min: 30, max: 200, required: true },
    height:   { type: Number, min: 100, max: 250, required: true },
    weightHistory: [{
        weight: Number,
        date: { type: Date, default: Date.now }
    }],
    createdAt: { type: Date, default: Date.now }
});

module.exports = mongoose.model('User', UserSchema);
```
