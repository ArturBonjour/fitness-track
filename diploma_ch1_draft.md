# ВВЕДЕНИЕ

Цифровизация сферы здоровья и спорта формирует устойчивый спрос на программные решения, которые помогают пользователю планировать физическую активность, контролировать прогресс и принимать решения на основе данных. При этом значительная часть существующих фитнес-сервисов либо перегружена второстепенными функциями, либо ориентирована на узкие сценарии использования. В результате у пользователей часто отсутствует простой и целостный инструмент, который объединяет планирование тренировок, контроль целей и визуальную аналитику в одном веб-приложении.

Актуальность темы обусловлена необходимостью разработки доступной информационной системы персонального фитнес-мониторинга, которая:
- поддерживает базовые процессы самоорганизации тренировок;
- снижает порог входа для регулярного ведения данных;
- предоставляет наглядный механизм контроля динамики результатов;
- обеспечивает расширяемую архитектуру для дальнейшего развития функциональности.

Проблемная область связана с тем, что пользовательские данные о тренировках и целях часто хранятся фрагментарно, а применяемые инструменты не обеспечивают достаточного баланса между простотой интерфейса и аналитическими возможностями. Практическая задача исследования заключается в проектировании и реализации полнофункционального веб-приложения фитнес-трекера с современной клиент-серверной архитектурой.

**Цель работы** — разработать веб-приложение «Bogdanov FitTrack», обеспечивающее учет тренировок, постановку и отслеживание целей, а также анализ пользовательского прогресса на основе структурированных данных.

Для достижения цели поставлены следующие задачи:
1. Проанализировать предметную область персонального фитнес-трекинга и выявить ключевые функциональные требования к системе.
2. Выполнить обзор существующих программных решений и определить ограничения, которые должны быть устранены в разрабатываемом продукте.
3. Сформировать требования к архитектуре, данным, безопасности и пользовательскому интерфейсу приложения.
4. Спроектировать структуру клиентской и серверной частей системы, включая модель взаимодействия компонентов и API.
5. Реализовать программный прототип системы с поддержкой авторизации, работы с тренировками и целями, а также визуализации прогресса.
6. Провести проверку работоспособности и оценить качество реализации на уровне основных пользовательских сценариев.

**Объект исследования** — процессы цифрового сопровождения персональной физической активности пользователя.

**Предмет исследования** — методы и средства проектирования и разработки веб-приложения фитнес-трекинга с модульной архитектурой, REST API и визуальной аналитикой.

Практическая значимость работы состоит в создании готовой программной основы, пригодной для дальнейшего внедрения и масштабирования: расширения справочников тренировок, интеграции с внешними устройствами и совершенствования аналитического блока.

Структурно работа включает введение, главы основной части, заключение, список использованных источников и приложения. В первой главе рассматриваются теоретические и прикладные аспекты предметной области, а также формулируются требования к проектируемой системе.

# 1 АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ТРЕБОВАНИЙ К СИСТЕМЕ ФИТНЕС-ТРЕКИНГА

## 1.1 Предметная область персонального фитнес-мониторинга

Персональный фитнес-мониторинг представляет собой совокупность процессов планирования, фиксации и анализа тренировочной активности пользователя. С практической точки зрения ядро предметной области включает:
- календарное планирование тренировок;
- учет параметров тренировочной сессии (тип, дата, длительность, комментарии);
- постановку измеримых целей с дедлайнами;
- оценку текущего прогресса по количественным показателям;
- поддерживающую аналитику (графики, агрегированные показатели).

Ключевая особенность данной области — цикличность поведения пользователя. В отличие от разовых задач, фитнес-активность требует длительного сопровождения, регулярного ввода данных и мотивационной обратной связи. Следовательно, программный продукт должен быть ориентирован не только на хранение информации, но и на снижение когнитивной нагрузки при ежедневном использовании.

Для рассматриваемой области важны следующие свойства информационной системы:
1. **Оперативность ввода данных** — пользователь должен фиксировать тренировку за минимальное количество действий.
2. **Согласованность данных** — изменения профиля, тренировок и целей должны отражаться в аналитике без расхождений.
3. **Наглядность прогресса** — система должна предоставлять визуальные инструменты интерпретации динамики.
4. **Надежность и безопасность** — персональные данные пользователя должны быть защищены механизмами аутентификации и контроля доступа.
5. **Расширяемость** — архитектура должна допускать развитие функционала без полного переписывания системы.

Таким образом, предметная область сочетает требования к удобству интерфейса, целостности данных и инженерной устойчивости программного решения.

## 1.2 Анализ существующих решений

На рынке цифровых фитнес-сервисов представлены как универсальные платформы, так и специализированные приложения. Большинство решений можно условно разделить на три группы:

1. **Системы общего трекинга здоровья**  
   Предоставляют широкий набор функций (питание, активность, сон), но зачастую усложняют базовый сценарий учета тренировок избыточным количеством параметров.

2. **Спортивно-ориентированные трекеры**  
   Акцентированы на определенных типах активности (бег, велоспорт, силовые программы), что ограничивает универсальность для пользователей со смешанным режимом тренировок.

3. **Легковесные дневники тренировок**  
   Обеспечивают простую фиксацию данных, но часто не содержат качественной аналитики, прогресс-метрик и полноценной серверной архитектуры.

По итогам анализа можно выделить типовые ограничения аналогов:
- недостаточная гибкость в управлении целями;
- слабая связность между календарным планированием и аналитическим блоком;
- ограниченная персонализация интерфейса;
- высокая зависимость от платных функций при базовых потребностях пользователя.

Следовательно, разработка собственного приложения «Bogdanov FitTrack» обоснована с точки зрения закрытия практического разрыва между простотой ежедневного использования и полнотой ключевого функционала.

## 1.3 Формирование требований к разрабатываемой системе

На основе анализа предметной области и существующих решений сформирован набор требований к проектируемому приложению.

### 1.3.1 Функциональные требования

Система должна обеспечивать:
1. Регистрацию и аутентификацию пользователя с выдачей токена доступа.
2. Просмотр и редактирование персонального профиля.
3. Создание, просмотр и удаление тренировок.
4. Календарное представление тренировок и привязанных событий.
5. Создание и сопровождение целей с параметрами прогресса и дедлайнами.
6. Получение и отображение рекомендаций по тренировкам.
7. Отображение графиков динамики веса и активности.
8. Уведомление пользователя о результатах операций и ошибках.

### 1.3.2 Нефункциональные требования

К системе предъявляются следующие нефункциональные требования:
- **Производительность:** интерфейс должен обеспечивать комфортную работу при типовом объеме пользовательских записей.
- **Надежность:** операции чтения/записи должны выполняться с контролем ошибок и корректной обратной связью.
- **Безопасность:** доступ к пользовательским данным осуществляется только после аутентификации; защищенные маршруты API должны проверять принадлежность ресурсов владельцу.
- **Поддерживаемость:** кодовая база должна быть модульной и разделенной на клиентскую и серверную части.
- **UX-качество:** интерфейс должен поддерживать адаптивность, визуальную иерархию, светлую/темную темы и понятные состояния загрузки.

### 1.3.3 Требования к архитектуре и технологическому стеку

Для реализации выбран клиент-серверный подход:
- **Frontend:** React-приложение с компонентной архитектурой, маршрутизацией и контекстами состояния.
- **Backend:** REST API на Node.js/Express с middleware-подходом к аутентификации и валидации.
- **Хранение данных:** MongoDB через Mongoose-модели с выделением сущностей пользователя, тренировки, цели и рекомендаций.
- **Интеграционный слой:** Axios-клиент для взаимодействия frontend с backend.

Такое разделение обеспечивает независимое развитие слоев, упрощает сопровождение проекта и соответствует современным практикам разработки веб-систем.

### 1.3.4 Ожидаемый результат первой главы

По результатам проведенного анализа определены:
- границы и специфика предметной области;
- ограничения существующих решений;
- функциональные и нефункциональные требования к системе;
- обоснованный архитектурно-технологический базис проекта.

Полученные положения формируют теоретико-методическую основу дальнейшего проектирования программного продукта.

## 2.1 Проектирование архитектуры системы

### 2.1.1 Общая архитектура

Архитектура системы «Bogdanov FitTrack» построена по принципу разделения ответственности между клиентской и серверной частями. Взаимодействие реализовано по схеме клиент — REST API — база данных (Рисунок 2.1).

Таблица 2.1 — Основные архитектурные уровни системы

- независимое масштабирование каждого уровня;

- возможность замены клиентского приложения (например, на мобильное) без изменения API.

## 2.2 Проектирование модели данных

### 2.2.1 Обоснование выбора MongoDB

Для хранения данных выбрана документно-ориентированная СУБД MongoDB. Решение обосновано следующими характеристиками предметной области:
- тренировки и цели имеют переменную структуру атрибутов, удобно представляемую в формате документа;
- история веса является встроенным массивом документов в рамках сущности пользователя;
- отсутствует необходимость в сложных транзакционных операциях между коллекциями;
- Mongoose ODM обеспечивает схемную валидацию данных на уровне приложения, компенсируя гибкость NoSQL контролируемыми ограничениями.

### 2.2.2 Описание коллекций

Система включает четыре основных коллекции MongoDB:

**Коллекция `users`** хранит данные пользователей и историю веса:

```json
{
  "_id": "ObjectId",
  "name": "String (required)",
  "email": "String (required, unique)",
  "password": "String (bcrypt, required)",
  "gender": "enum: [male, female]",
  "age": "Number (15–100)",
  "weight": "Number (30–200)",
  "height": "Number (100–250)",
  "weightHistory": [
    { "weight": "Number", "date": "Date" }
  ],
  "createdAt": "Date"
}
```

**Коллекция `workouts`** хранит тренировочные записи пользователя:

```json
{
  "_id": "ObjectId",
  "user": "ObjectId → users",
  "type": "String (required)",
  "date": "Date (required)",
  "time": "String (HH:mm, required)",
  "duration": "Number (минуты, required)",
  "comment": "String (optional)",
  "createdAt": "Date"
}
```

**Коллекция `goals`** хранит фитнес-цели пользователей с прогрессом:

```json
{
  "_id": "ObjectId",
  "user": "ObjectId → users",
  "title": "String (required)",
  "type": "String (required)",
  "startValue": "Number (required)",
  "currentValue": "Number (required)",
  "targetValue": "Number (required)",
  "unit": "String (default: 'кг')",
  "deadline": "Date (required)",
  "description": "String (optional)",
  "completed": "Boolean (default: false)",
  "createdAt": "Date"
}
```

**Коллекция `recommendations`** хранит шаблоны рекомендаций по тренировкам:

```json
{
  "_id": "ObjectId",
  "goal": "String (weightLoss | muscleGain | endurance)",
  "level": "String (beginner | intermediate | advanced)",
  "frequency": "String",
  "duration": "String",
  "types": ["String"],
  "tips": ["String"]
}
```

### 2.2.3 Диаграмма связей коллекций

На уровне ссылок между документами применяется подход «ссылка по ObjectId»: коллекции `workouts` и `goals` ссылаются на `users` через поле `user`. История веса реализована как встроенный массив внутри документа `users`, что позволяет загружать её за одно обращение к базе данных без дополнительного join.

```
users
  ├── _id ───────┐
  │              │ ref
  ├── ...        ├─── workouts.user
  └── weightHist │    goals.user
                 └────────────────
```

## 2.3 Проектирование REST API

### 2.3.1 Структура API

API реализован по архитектурному стилю REST и использует следующие соглашения:
- базовый путь: `/api`;
- аутентификация: JWT-токен передаётся в заголовке `Authorization: Bearer <token>`;
- формат данных: JSON;
- HTTP-методы соответствуют семантике CRUD: GET — чтение, POST — создание, PUT — обновление, DELETE — удаление.

### 2.3.3 Механизм аутентификации

Аутентификация реализована на основе JSON Web Tokens (JWT). Схема работы:

1. Пользователь отправляет POST /api/auth/login с email и паролем.
2. Сервер проверяет пароль (bcrypt), генерирует JWT с полезной нагрузкой `{ user: { id } }` и сроком действия 24 часа.
3. Клиент сохраняет токен в localStorage и прикрепляет его к каждому запросу через заголовок `Authorization: Bearer <token>`.
4. Middleware `auth.js` верифицирует токен при каждом обращении к защищённому маршруту и помещает идентификатор пользователя в `req.user.id`.
5. При выходе клиент удаляет токен; серверная сторона не требует дополнительных действий (JWT — stateless).

### 2.3.4 Валидация данных на сервере

Валидация входных данных реализована через библиотеку `express-validator`. Для каждого маршрута, принимающего тело запроса, задан набор правил. Пример правил для создания тренировки:

```javascript
[
  check('type', 'Тип тренировки обязателен').not().isEmpty(),
  check('date', 'Дата обязательна').isISO8601(),
  check('time', 'Время обязательно').matches(/^([01]\d|2[0-3]):([0-5]\d)$/),
  check('duration', 'Продолжительность от 1 до 600 мин').isInt({ min: 1, max: 600 })
]
```

При наличии ошибок сервер возвращает ответ `HTTP 400` с массивом ошибок в формате JSON.

## 3.2 Реализация серверной части

### 3.2.1 Модуль аутентификации

Аутентификация реализована по модели Stateless JWT. При регистрации (`POST /api/auth/register`) сервер:
1. Валидирует все входные поля через массив правил `express-validator`.
2. Проверяет уникальность email в коллекции `users`.
3. Хэширует пароль алгоритмом bcrypt с факторами соли 10 раундов: `await bcrypt.hash(password, salt)`.
4. Сохраняет документ пользователя в MongoDB, включая начальное значение истории веса.
5. Возвращает JWT-токен со сроком действия 7 дней вместе с профилем пользователя.

При авторизации (`POST /api/auth/login`) дополнительно применяется rate-limiter (`express-rate-limit`): не более 5 попыток за 15 минут с одного IP. Это защищает от атак методом перебора.

Листинг 3.2 — Middleware JWT-верификации (`middleware/auth.js`)

```javascript
module.exports = function (req, res, next) {
    let token = req.header('x-auth-token') || req.header('Authorization');
    if (token && token.startsWith('Bearer ')) token = token.slice(7);
    if (!token) return res.status(401).json({ message: 'Нет токена, авторизация отклонена' });

    try {
        const decoded = jwt.verify(token, process.env.JWT_SECRET);
        req.user = decoded.user;
        next();
    } catch (err) {
        res.status(401).json({ message: 'Токен недействителен' });
    }
};
```

### 3.2.2 Модуль управления тренировками

Маршруты тренировок организованы с применением паттерна «ресурс + вложенные фильтры». Три специализированных GET-маршрута (`/date/:date`, `/range/:start/:end`, `/:id`) объявлены перед общим маршрутом `/:id`, что обеспечивает корректную диспетчеризацию Express-роутера.

Алгоритм создания тренировки (`POST /api/workouts`):
1. Проверка JWT через middleware `auth`.
2. Валидация полей: тип, дата, время, длительность.
3. Создание объекта `Workout` с привязкой к идентификатору текущего пользователя из `req.user.id`.
4. Сохранение документа в MongoDB, возврат сохранённого объекта.

Листинг 3.3 — Создание тренировки

```javascript
const newWorkout = new Workout({
    user: req.user.id,
    type, date, time, duration, comment
});
const workout = await newWorkout.save();
res.json(workout);
```

Операция удаления использует метод `.deleteOne()` (что соответствует актуальной версии Mongoose) после проверки принадлежности ресурса текущему пользователю. Проверка принадлежности (`workout.user.toString() !== req.user.id`) обеспечивает изоляцию данных между пользователями.

### 3.2.3 Модуль управления целями

Управление целями включает специализированный маршрут `PUT /:id/progress`, реализующий алгоритм автоматической проверки достижения цели:

```javascript
if (goal.type === 'вес' || goal.type === 'сила') {
    if (goal.startValue < goal.targetValue) {
        goal.completed = goal.currentValue >= goal.targetValue;
    } else {
        goal.completed = goal.currentValue <= goal.targetValue;
    }
}
```

Данный алгоритм определяет направление цели (на увеличение или уменьшение) и автоматически переводит её в статус «выполнена» при достижении целевого значения.

### 3.2.4 История веса пользователя

История весовых показателей реализована через встроенный массив `weightHistory` внутри документа пользователя. При обновлении профиля (`PUT /api/users/profile`) новый вес сравнивается с текущим. При расхождении значений в массив добавляется новая запись с текущей временной меткой:

```javascript
if (weight !== undefined && weight !== user.weight) {
    user.weight = weight;
    user.weightHistory.push({ weight, date: Date.now() });
}
```

Эндпоинт `GET /api/users/weight-history` возвращает последние 30 записей, что достаточно для отображения наглядного графика динамики.

## 3.3 Реализация клиентской части

### 3.3.1 Настройка Axios и перехватчик запросов

Взаимодействие клиента с API осуществляется через библиотеку Axios. Кастомный хук `useAxiosInterceptor` настраивает перехватчик ответов: при получении HTTP 401 токен удаляется из localStorage и пользователь перенаправляется на страницу входа. Это обеспечивает единое место обработки истечения сессии без дублирования логики в каждом запросе.

```javascript
axios.interceptors.response.use(
    (response) => response,
    (error) => {
        if (error.response?.status === 401) {
            localStorage.removeItem('token');
            window.location.href = '/login';
        }
        return Promise.reject(error);
    }
);
```

### 3.3.2 Контекст аутентификации (AuthContext)

`AuthContext` — центральный источник истины о состоянии сессии. При монтировании приложения хук `useEffect` проверяет наличие токена в `localStorage` и загружает профиль пользователя через `GET /api/auth/user`. При успехе состояние `isAuthenticated` устанавливается в `true` и пользователь не видит экраны входа/регистрации.

Метод `updateUser` обновляет данные профиля на сервере и синхронизирует локальное состояние, исключая необходимость перезагрузки страницы.

### 3.3.3 Компонент Calendar

Компонент `Calendar` — ключевой элемент главного экрана. Он реализует следующую логику:

1. **Генерация сетки**: вычисляются дни текущего месяца с учётом начального дня недели; добавляются «пустые» ячейки для выравнивания первой строки.
2. **Загрузка данных**: при смене месяца выполняется запрос `GET /api/workouts/range/:start/:end` для получения тренировок за период.
3. **Отображение бейджей**: тренировки и цели, привязанные к дате, отображаются цветными метками в ячейке соответствующего дня.
4. **Добавление записей**: клик по пустой ячейке открывает модальное окно с формой создания тренировки или цели, предзаполненной выбранной датой.

### 3.3.4 Компонент Profile

Страница профиля объединяет несколько функциональных блоков:
- **Блок информации**: отображение и редактирование персональных данных.
- **Граф динамики веса**: интерактивный линейный график `react-chartjs-2` с данными из истории весовых показателей.
- **Граф активности**: недельная агрегация тренировок для визуализации частоты занятий.
- **Список тренировок**: фильтруемый по дате и типу список с ключевыми параметрами.
- **Список целей**: карточки с прогрессом и статусом выполнения.

### 3.3.5 Компонент WorkoutRecommendations

Компонент загружает персонализированные рекомендации через `GET /api/recommendations/:goal/:level` с использованием React `useEffect`. Параметры (цель и уровень подготовки) управляются локальным состоянием. При изменении любого из параметров эффект автоматически повторно запрашивает данные. Во время загрузки отображается Skeleton-плейсхолдер; при ошибке — блок с кнопкой повторной попытки через счётчик `retryCount`.

### 3.3.6 Калькулятор калорий

Компонент `CalorieCalculator` реализует расчёт суточного потребления калорий по формуле Харриса-Бенедикта:

- Для мужчин: `BMR = 88.362 + 13.397 × W + 4.799 × H − 5.677 × A`
- Для женщин: `BMR = 447.593 + 9.247 × W + 3.098 × H − 4.330 × A`

Где W — масса тела (кг), H — рост (см), A — возраст (лет). Суточная норма калорий (TDEE) рассчитывается умножением BMR на коэффициент физической активности (от 1.2 до 1.9). Целевое потребление корректируется в зависимости от выбранной цели (похудение: ×0.85, поддержание: ×1.0, набор: ×1.15).

В интерфейсе реализованы следующие UX-паттерны:

**Skeleton-загрузка**: вместо спиннеров при запросах данных отображаются анимированные серые блоки формы ожидаемого контента. Это снижает воспринимаемое время ожидания.

**Toast-уведомления**: всплывающие уведомления появляются в правом нижнем углу с анимацией `slideInRight`. Каждый тост содержит иконку типа события, сообщение и прогресс-бар, визуально отображающий оставшееся время показа.

**Модальные окна**: реализованы через `ReactDOM.createPortal` в контейнер `#modal-root`, что исключает конфликты z-index с основным деревом DOM. При открытии — анимация `bounceIn`; фон блокируется `overflow: hidden`.

**Типы тренировок и целей как пиктограммы**: вместо стандартного `<select>` применяется сетка кнопок-пилюль с эмодзи и цветовым акцентом, что ускоряет выбор и делает интерфейс более визуально выразительным.

**Индикатор силы пароля**: при вводе пароля на экране регистрации в реальном времени отображается 4-уровневый индикатор силы (Слабый / Нормальный / Хороший / Отличный) с цветовой кодировкой.

## 3.5 Руководство пользователя

### 3.5.1 Регистрация и вход

Для начала работы с приложением пользователь переходит по адресу приложения в браузере. На экране входа предлагается ввести email и пароль зарегистрированной учётной записи. При первом посещении пользователь нажимает кнопку «Зарегистрироваться» и заполняет форму с именем, email, паролем (минимум 6 символов), полом, возрастом, весом и ростом. После успешной регистрации пользователь автоматически аутентифицируется и переходит на главную страницу.

### 3.5.2 Управление тренировками

На главной странице отображается интерактивный календарь. Для добавления тренировки пользователь кликает по нужному дню и выбирает «Добавить тренировку». В открывшемся модальном окне выбирается тип тренировки из пиктограмм (Силовая / Кардио / Растяжка / Йога / Плавание / Другое), устанавливается дата, время и продолжительность с помощью ползунка (от 5 до 180 минут). Тренировка сохраняется и отображается в ячейке календаря фиолетовым бейджем.

### 3.5.3 Управление целями

Для создания цели в модальном окне указывается название, тип цели (Вес / Частота / Дистанция / Сила / Другое), начальное и целевое значения, дедлайн. Форма отображает предпросмотр прогресса в виде зелёной полосы. После сохранения цель отображается на странице профиля с актуальным прогрессом.

### 3.5.4 Просмотр профиля и аналитики

Страница профиля доступна через ссылку «Профиль» в навигации. Она содержит:
- Персональные данные с возможностью редактирования через кнопку «Редактировать».
- Линейный график динамики веса по датам обновления профиля.
- График недельной активности (количество тренировок по неделям).
- Фильтруемый список тренировок с выбором периода и типа.
- Список целей с прогресс-барами и статусами.

### 3.5.5 Рекомендации и калькулятор

На главной странице справа расположены два инструмента:
- **Калькулятор калорий**: автоматически рассчитывает базовый обмен веществ, суточную норму и целевое потребление по данным профиля. Пользователь выбирает уровень активности и цель (похудение / поддержание / набор).
- **Рекомендации по тренировкам**: подбираются персонализированные советы по частоте, длительности и типам тренировок в зависимости от цели и уровня подготовки.

Проведён анализ предметной области персонального фитнес-мониторинга: выявлены ключевые сценарии использования, изучены существующие решения (Strava, MyFitnessPal, Google Fit), определены их достоинства и недостатки. На основе анализа сформулированы функциональные и нефункциональные требования к разрабатываемой системе.

2. Обоснован технологический стек системы: Node.js/Express для серверной части, React/Tailwind CSS для клиентской части, MongoDB в качестве хранилища данных. Обоснован выбор документно-ориентированной СУБД, JWT-аутентификации и компонентной архитектуры.

3. Разработана трёхуровневая клиент-серверная архитектура. Спроектированы: документно-ориентированная модель данных с четырьмя коллекциями, спецификация REST API с 20 маршрутами, иерархия React-компонентов и модель глобального состояния на Context API. Построены UML-диаграммы вариантов использования и последовательности.

4. Реализованы все компоненты системы: серверная часть с JWT-аутентификацией, bcrypt-хэшированием паролей, rate-limiting, серверной валидацией и полным CRUD для тренировок и целей; клиентская часть с интерактивным календарём, профилем с Chart.js-графиками, рекомендациями и калькулятором калорий.

5. Реализован «wow»-уровень пользовательского интерфейса: glassmorphism-карточки аутентификации, анимированные модальные окна, skeleton-загрузка, пиктограммный выбор типов, индикатор силы пароля, toast-уведомления с прогресс-баром, тёмная тема, адаптивная навигация с плавным раскрытием на мобильных устройствах.

6. Проведено функциональное тестирование (25 тест-кейсов API) и тестирование пользовательского интерфейса (21 тест-кейс). Все тест-кейсы завершились успешно. Производственная сборка проходит без ошибок.

Разработанная система обеспечивает полный цикл работы пользователя с фитнес-данными: от регистрации и планирования тренировок до визуализации прогресса и получения персонализированных рекомендаций. Интерфейс приложения соответствует современным стандартам UX/UI: поддержка тёмной темы, адаптивность на всех типах устройств, мгновенная обратная связь при пользовательских действиях.

Направлениями дальнейшего развития системы могут служить: интеграция с носимыми устройствами (Fitbit, Apple Watch) через соответствующие API; добавление социальных функций (друзья, публичные достижения); реализация Push-уведомлений через Service Worker для напоминания о запланированных тренировках; переход на мобильное приложение на базе React Native с повторным использованием бизнес-логики.

const checkAuth = async () => {
    const token = localStorage.getItem('token');
    if (token) {
        axios.defaults.headers.common['Authorization'] = `Bearer ${token}`;
        const response = await axios.get('/api/auth/user');
        setUser(response.data);
        setIsAuthenticated(true);
    }
};

const login = async (userData) => {
    const response = await axios.post('/api/auth/login', userData);
    const { token, user } = response.data;
    localStorage.setItem('token', token);
    axios.defaults.headers.common['Authorization'] = `Bearer ${token}`;
    setUser(user); setIsAuthenticated(true);
    return user;
};
