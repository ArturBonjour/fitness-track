# ВВЕДЕНИЕ

Цифровизация сферы здоровья и спорта формирует устойчивый спрос на программные решения, которые помогают пользователю планировать физическую активность, контролировать прогресс и принимать решения на основе данных. Согласно исследованиям Grand View Research, мировой рынок мобильных приложений для здоровья и фитнеса оценивается в 14,7 млрд долларов США и продолжает расти ежегодно на 17,6% [1]. При этом значительная часть существующих сервисов либо перегружена второстепенными функциями, либо ориентирована на узкие сценарии использования.

Степень разработанности темы в области отечественных исследований остается невысокой: большинство российских разработчиков применяют готовые зарубежные платформы, не адаптированные к специфике отечественного пользователя. Зарубежные решения — Strava, MyFitnessPal, Google Fit — занимают лидирующие позиции, однако не предоставляют комплексного инструмента, объединяющего планирование тренировок, контроль целей и визуальную аналитику в одном легковесном веб-приложении [2].

Проблемная область связана с тем, что пользовательские данные о тренировках и целях часто хранятся фрагментарно, а применяемые инструменты не обеспечивают достаточного баланса между простотой интерфейса и аналитическими возможностями. Практическая задача данной работы состоит в проектировании и реализации полнофункционального веб-приложения фитнес-трекера с современной клиент-серверной архитектурой.

**Цель работы** — разработать информационную систему персонального фитнес-мониторинга «Bogdanov FitTrack», обеспечивающую учёт тренировок, постановку и отслеживание фитнес-целей, а также визуализацию пользовательского прогресса на основе структурированных данных, хранимых на сервере.

Для достижения поставленной цели необходимо решить следующие задачи.

1. Выявить ключевые функциональные паттерны персонального фитнес-мониторинга путём анализа предметной области и сравнительного обзора ведущих программных аналогов.
2. Определить функциональные и нефункциональные требования к разрабатываемой системе, учитывающие ограничения существующих решений.
3. Обосновать архитектурный подход и технологический стек на основе анализа требований, современных практик веб-разработки и специфики задачи.
4. Спроектировать структуру базы данных, REST API и компонентную иерархию клиентской части с использованием UML-диаграмм.
5. Реализовать серверный и клиентский модули системы с поддержкой аутентификации, полного цикла управления тренировками и целями, визуализации прогресса и персонализированных рекомендаций.
6. Провести функциональное тестирование ключевых сценариев использования и подтвердить корректность реализованных компонентов.

**Объект исследования** — процессы цифрового сопровождения персональной физической активности пользователя.

**Предмет исследования** — методы и средства проектирования и разработки веб-приложения фитнес-мониторинга с модульной клиент-серверной архитектурой, REST API и визуальной аналитикой.

Практическая значимость работы состоит в создании готовой программной основы, пригодной для внедрения и масштабирования: расширения базы тренировок, интеграции с носимыми устройствами и совершенствования аналитического блока.

Структурно работа включает введение, четыре главы основной части, заключение, список использованных источников и приложение А с ключевыми листингами кода. В первой главе рассматриваются теоретические и прикладные аспекты предметной области, формулируются требования. Во второй главе описывается проектирование системы. В третьей главе представлена программная реализация. В четвёртой главе проводится тестирование.

# 1 АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ТРЕБОВАНИЙ К СИСТЕМЕ ФИТНЕС-ТРЕКИНГА

## 1.1 Предметная область персонального фитнес-мониторинга

Персональный фитнес-мониторинг представляет собой совокупность процессов планирования, фиксации и анализа тренировочной активности пользователя. В соответствии с классификацией Warburton et al. [3] регулярная физическая активность является одним из наиболее эффективных средств профилактики хронических заболеваний. Цифровые инструменты фитнес-трекинга формируют информационную основу для применения данного принципа на практике.

С технической точки зрения ядро предметной области включает следующие процессы:
- календарное планирование тренировок;
- учёт параметров тренировочной сессии (тип, дата, длительность, комментарии);
- постановку измеримых целей с дедлайнами;
- оценку текущего прогресса по количественным показателям;
- поддерживающую аналитику (графики, агрегированные показатели);
- персонализированные рекомендации на основе профиля и цели пользователя.

Ключевая особенность данной области — цикличность поведения пользователя. В отличие от разовых задач, фитнес-активность требует длительного сопровождения, регулярного ввода данных и мотивационной обратной связи. Следовательно, программный продукт должен быть ориентирован не только на хранение информации, но и на снижение когнитивной нагрузки при ежедневном использовании [4].

Для рассматриваемой области существенны следующие свойства информационной системы.

1. **Оперативность ввода данных** — пользователь должен фиксировать тренировку за минимальное количество действий.
2. **Согласованность данных** — изменения профиля, тренировок и целей должны немедленно отражаться в аналитике.
3. **Наглядность прогресса** — система должна предоставлять визуальные инструменты для интерпретации динамики показателей.
4. **Надёжность и безопасность** — персональные данные должны быть защищены механизмами аутентификации и авторизации.
5. **Расширяемость** — архитектура должна допускать развитие функционала без переработки ядра системы.

Таким образом, предметная область сочетает требования к удобству интерфейса, целостности данных и инженерной устойчивости программного решения.

- высокая зависимость REST API от платных тарифов при базовых потребностях разработчика.

Следовательно, разработка собственного приложения «Bogdanov FitTrack» обоснована с точки зрения закрытия практического разрыва между простотой ежедневного использования и полнотой ключевого функционала.

## 1.3 Формирование требований к разрабатываемой системе

На основе анализа предметной области и существующих решений сформирован набор требований к проектируемому приложению.

### 1.3.1 Функциональные требования

Система должна обеспечивать:
- регистрацию и аутентификацию пользователей с хранением персональных физических параметров;
- создание, просмотр, редактирование и удаление записей о тренировках;
- визуализацию тренировок в виде интерактивного месячного календаря;
- постановку фитнес-целей с начальным и целевым значениями, единицей измерения и дедлайном;
- автоматическое вычисление процента достижения цели и изменение статуса на «выполнена»;
- отображение графика динамики веса пользователя;
- агрегированную статистику: количество тренировок за неделю/месяц, серия дней, ИМТ;
- расчёт суточной нормы калорий по формуле Харриса-Бенедикта с учётом уровня активности и цели;
- персонализированные рекомендации по тренировкам в зависимости от цели и уровня подготовки.

### 1.3.3 Требования к архитектуре и технологическому стеку

Анализ требований и современных практик веб-разработки позволяет обосновать следующий технологический стек.

Для серверной части выбран Node.js в сочетании с фреймворком Express.js [15]. Данное решение обеспечивает неблокирующую I/O-модель, широкую экосистему пакетов и простоту разработки REST API [8]. В качестве базы данных выбрана MongoDB с ODM-библиотекой Mongoose [16]: документно-ориентированная модель хранения наиболее органична для разнородных данных пользователей (профиль, история веса, тренировки, цели) [9].

Клиентская часть реализована на React — библиотеке декларативного построения UI-компонентов [10]. Навигация между страницами организована с применением React Router v6 [20]. Управление глобальным состоянием осуществляется через встроенный Context API без дополнительных зависимостей. Стилизация выполнена с применением Tailwind CSS — утилитарного фреймворка с поддержкой адаптивных префиксов и тёмной темы через стратегию `class` [11].

### 1.3.4 Выводы по первой главе

Анализ предметной области и существующих решений показал, что ни один из рассмотренных аналогов не предоставляет в бесплатном режиме полный комплекс функций, необходимых для персонального фитнес-мониторинга. Сформированные функциональные и нефункциональные требования, а также обоснованный технологический стек создают основу для проектирования системы «Bogdanov FitTrack», описанного во второй главе.

## 2.2 Проектирование модели данных

### 2.2.1 Обоснование выбора MongoDB

Выбор документно-ориентированной СУБД MongoDB обоснован следующими соображениями. Во-первых, данные пользователя (профиль, история веса) образуют естественное дерево документов. Во-вторых, схемы тренировок и целей могут расширяться без миграций: MongoDB позволяет добавлять поля без изменения существующих документов [9]. В-третьих, в паре с ODM Mongoose достигается строгая схема на уровне приложения при сохранении гибкости базы данных.

## 3.2 Реализация серверной части

### 3.2.1 Модуль аутентификации

Аутентификация реализована по модели Stateless JWT. При регистрации сервер: валидирует все входные поля через массив правил express-validator; проверяет уникальность email в коллекции `users`; хэширует пароль алгоритмом bcrypt с 10 раундами соли; сохраняет документ пользователя; возвращает JWT-токен со сроком 7 дней.

При входе применяется rate-limiter на базе библиотеки express-rate-limit [22]: не более 5 попыток за 15 минут с одного IP. Данный механизм защиты соответствует рекомендациям OWASP по предотвращению атак перебора паролей [18]. На рисунке 3.2 представлена реализация middleware JWT-верификации.

```javascript
// middleware/auth.js
module.exports = function (req, res, next) {
    let token = req.header('x-auth-token') || req.header('Authorization');
    if (token && token.startsWith('Bearer ')) token = token.slice(7);
    if (!token) return res.status(401).json({ message: 'Нет токена' });
    try {
        const decoded = jwt.verify(token, process.env.JWT_SECRET);
        req.user = decoded.user;
        next();
    } catch (err) {
        res.status(401).json({ message: 'Токен недействителен' });
    }
};
```

При обновлении профиля новый вес сравнивается с текущим. При расхождении значений в массив `weightHistory` добавляется новая запись с временной меткой. Эндпоинт `GET /api/users/weight-history` возвращает последние 30 записей для построения графика.

### 3.3.1 Настройка Axios и перехватчик запросов

Хук `useAxiosInterceptor` настраивает перехватчик ответов Axios: при получении HTTP 401 токен удаляется из localStorage и пользователь перенаправляется на страницу входа. Это обеспечивает единое место обработки истечения сессии без дублирования логики в каждом компоненте.

### 3.3.2 Компонент StatsWidget

Компонент `StatsWidget` — ключевая новая функция, аналогичная сводным дашбордам Strava и Nike Training Club. При монтировании загружает все тренировки пользователя и вычисляет следующие показатели.

1. **Тренировок за неделю** и суммарные минуты — фильтрация по `dayjs().startOf('isoWeek')` [23].
2. **Тренировок за месяц** и суммарные минуты — фильтрация по `dayjs().startOf('month')`.
3. **Серия дней (streak)** — обратный обход от сегодня по множеству дат (структура данных Set языка JavaScript [21]) с тренировками до первого пропуска.
4. **Любимый тип** тренировки за месяц — агрегация по полю `type` с выбором максимума.
5. **ИМТ** — вычисляется из данных профиля: `weight / (height/100)²`, с категоризацией по стандарту ВОЗ [13].

На рисунке 3.5 представлен алгоритм вычисления серии дней.

```javascript
// StatsWidget.js (фрагмент)
const datesSet = new Set(
    all.map(w => dayjs(w.date).format('YYYY-MM-DD'))
);
let streak = 0;
let d = dayjs();
while (datesSet.has(d.format('YYYY-MM-DD'))) {
    streak++;
    d = d.subtract(1, 'day');
}
```

## 3.5 Руководство пользователя

### 3.5.1 Регистрация и вход

Для начала работы с приложением пользователь переходит по адресу приложения в браузере. На экране входа вводится email и пароль. При первом посещении нажимается кнопка «Зарегистрироваться»; заполняется форма с именем, email, паролем, полом, возрастом, весом и ростом. При вводе пароля отображается цветной индикатор силы (Слабый / Нормальный / Хороший / Отличный). После успешной регистрации происходит автоматический вход и переход на главную страницу.

На главной странице отображается интерактивный календарь. Для добавления тренировки следует кликнуть по нужному дню. В открывшемся модальном окне выбирается тип тренировки из пиктограмм, дата, время и продолжительность (ползунком 5–180 минут). После сохранения тренировка отображается в ячейке фиолетовым бейджем.

Для создания цели указывается название, тип (Вес / Частота / Дистанция / Сила / Другое), начальное и целевое значения, дедлайн. Форма отображает предпросмотр прогресса в виде зелёной полосы. После сохранения цель отображается на странице профиля с актуальным процентом достижения.

### 3.5.4 Просмотр профиля и аналитики

Страница «Профиль» содержит: персональные данные (кнопка «Редактировать»); линейный график динамики веса; столбчатый график недельной активности, реализованные с применением библиотеки Chart.js [19]; фильтруемый список тренировок с emoji-пиктограммами типов активности; список целей с прогресс-барами и индикатором срока. На главной странице виджет «Моя статистика» показывает тренировки за неделю/месяц, серию дней, ИМТ и любимый тип тренировки.

### 3.5.5 Рекомендации и калькулятор

Правая колонка главной страницы содержит: виджет статистики (StatsWidget), калькулятор калорий (BMR, TDEE, целевой расчёт) и блок персонализированных рекомендаций с выбором цели и уровня подготовки.

## А.1 Схема коллекции users (server/models/User.js)

```javascript
const mongoose = require('mongoose');

const UserSchema = new mongoose.Schema({
    name:     { type: String, required: true },
    email:    { type: String, required: true, unique: true },
    password: { type: String, required: true },
    gender:   { type: String, enum: ['male', 'female'], required: true },
    age:      { type: Number, min: 15, max: 100, required: true },
    weight:   { type: Number, min: 30, max: 200, required: true },
    height:   { type: Number, min: 100, max: 250, required: true },
    weightHistory: [{
        weight: Number,
        date: { type: Date, default: Date.now }
    }],
    createdAt: { type: Date, default: Date.now }
});

module.exports = mongoose.model('User', UserSchema);
```

const checkAuth = async () => {
    const token = localStorage.getItem('token');
    if (token) {
        axios.defaults.headers.common['Authorization'] = `Bearer ${token}`;
        const response = await axios.get('/api/auth/user');

    const response = await axios.post('/api/auth/login', userData);
    const { token, user } = response.data;
    localStorage.setItem('token', token);

};
